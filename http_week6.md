# HTTP란?

: _Hyper Text Transfer Protocol의 약자로 웹 브라우저(Client)와 서버가 HTML, 이미지 영상 등, 하이퍼미디어 문서를 주고받기 위해 사용되는 핵심 애플리케이션 계층 **통신 규약(프로토콜)**이다._

> - HTTP는 클라이언트 시스템에서 서버에 요청한 후, 서버에서 응답 메세지를 보내는 흐름.
> - HTTP는 기본적으로 TCP/IP 프로토콜을 기반으로 동작한다.

## HTTP의 특징

### 무상태성(Stateless)

HTTP는 **상태를 유지하지 않는 Stateless 프로토콜**이다.

- 각각의 요청을 별개의 것으로 처리
- 이전 요청의 정보를 저장하지 않는다.
  > - 서버 부하를 줄이고, 클라이언트와 서버 간의 통신을 단순화. 속도 향상.
  > - but 쿠키/세션/토큰을 사용해 로그인 상태 등을 유지한다.

### 비연결성(Connectionless)

HTTP는 **요청과 응답이 한 번 이루어지면 연결을 끊는 Connectionless 프로토콜**이다.

- 이전의 요청과 응답 정보 유지x
  > - 서버 부하를 줄일 수 있다.
  > - 새로운 연결을 맺는 데 시간이 추가로 발생할 수 있다.
  >   :_성능 향상을 위해 완전히 끊지 않고 재사용._

### 클라이언트 서버 구조(Client-Server Architecture)

HTTP는 **Request/Response 구조로 통신**한다.

- 클라이언트: 서버에 **요청(Request)**을 보냄.
- 서버: 클라이언트에게 **응답(Response)**을 보냄.
  > 이러한 요청과 응답은 HTTP 메세지 형식으로 주고받는다.

### 기타 특징

- HTTP, HTTP 메세지는 단순하다.
- 확장 가능한 기술이다.

## HTTP / HTTPS 차이점

| 구분   | HTTP             | HTTPS                   |
| ------ | ---------------- | ----------------------- |
| 포트   | 80               | 443                     |
| 보안   | 없음 (평문 전송) | SSL/TLS 암호화          |
| 속도   | 상대적으로 빠름  | 암호화로 인해 약간 느림 |
| 인증서 | 불필요           | SSL 인증서 필요         |

- HTTPS는 HTTP에 **SSL/TLS(보안 계층)** 를 추가한 프로토콜이다.
- 데이터를 암호화하여 전송하기 때문에 중간에서 데이터를 가로채도 내용을 알 수 없다.
- HTTPS는 초기 연결 시 비대칭키 기반의 핸드셰이크를 통해 대칭키를 생성하여 이후 데이터를 암호화한다.

## 요청 / 응답 구조

### Request (요청)

:클라이언트가 서버에 보내는 메세지

| 구성 요소      | 설명                                          |
| -------------- | --------------------------------------------- |
| **Start Line** | HTTP 메서드, 요청 URL, HTTP 버전              |
| **Headers**    | 요청에 대한 부가 정보 (Host, Content-Type 등) |
| **Body**       | 전송할 데이터 (POST, PUT 등에서 사용)         |

### Response (응답)

:서버가 클라이언트의 요청에 대해 보내는 메세지

| 구성 요소       | 설명                                    |
| --------------- | --------------------------------------- |
| **Status Line** | HTTP 버전, 상태 코드, 상태 메세지       |
| **Headers**     | 응답에 대한 부가 정보 (Content-Type 등) |
| **Body**        | 실제 응답 데이터 (HTML, JSON 등)        |

## HTTP Method

| 메서드   | 역할                         | 특징                                  |
| -------- | ---------------------------- | ------------------------------------- |
| `GET`    | 리소스 조회                  | Body 없이 URL로 데이터 전달, 멱등성 O |
| `POST`   | 요청 데이터 처리 (주로 생성) | Body에 데이터 포함, 멱등성 X          |
| `PUT`    | 리소스 전체 대체             | 없으면 새로 생성, 멱등성 O            |
| `PATCH`  | 리소스 부분 수정             | PUT과 달리 일부만 변경                |
| `DELETE` | 리소스 삭제                  | 멱등성 O                              |

> **멱등성(Idempotent)**: 같은 요청을 여러 번 해도 결과가 동일한 성질.  
> ex) GET으로 같은 데이터를 10번 조회해도 서버 상태는 변하지 않는다.

## 상태 코드

### 2XX (성공)

| 코드  | 의미       | 설명                              |
| ----- | ---------- | --------------------------------- |
| `200` | OK         | 요청 성공                         |
| `201` | Created    | 리소스 생성 성공 (주로 POST 응답) |
| `204` | No Content | 성공했지만 반환할 데이터 없음     |

### 4XX (클라이언트 오류)

| 코드  | 의미         | 설명                             |
| ----- | ------------ | -------------------------------- |
| `400` | Bad Request  | 잘못된 요청 (문법 오류 등)       |
| `401` | Unauthorized | 인증되지 않음 (로그인 필요)      |
| `403` | Forbidden    | 권한 없음 (로그인해도 접근 불가) |
| `404` | Not Found    | 요청한 리소스를 찾을 수 없음     |

> `401` / `403` 차이:
>
> - `401`: **누구시죠??** -> 인증(Authentication) 필요
> - `403`: **당신은 안됩니다!** -> 권한(Authorization) 없음

### 5XX (서버 오류)

| 코드  | 의미                  | 설명                            |
| ----- | --------------------- | ------------------------------- |
| `500` | Internal Server Error | 서버 내부 오류                  |
| `502` | Bad Gateway           | 게이트웨이/프록시 오류          |
| `503` | Service Unavailable   | 서버 일시적 과부하 또는 점검 중 |
